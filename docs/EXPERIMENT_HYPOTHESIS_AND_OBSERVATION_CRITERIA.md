# Experiment Hypothesis and Observation Criteria

## Status

This is a human/project evaluation document for Lyle and Project Anam.

It is not runtime guidance, not `soul.md`, not behavioral guidance, not a prompt, and not model configuration. It must not be loaded into runtime prompts or used to constrain the entity's personality, name, avatar, values, or fixed identity.

Entityhood is not assumed as fact. Entityhood or development through continuity is the experimental question.

## Purpose

This document defines what Project Anam is trying to observe before and after go-live.

It exists to:

- avoid moving goalposts after the system begins accumulating experience
- separate interesting outputs from evidence of development
- define observation windows
- establish a baseline comparison plan
- keep project evaluation separate from runtime instruction

The aim is not to force a personality or decide in advance what success must look like. The aim is to make observations more disciplined.

## Core Hypothesis

Project Anam tests whether persistent source-labeled memory, reflection artifacts, bounded research loops, source traces, open loops, and temporal continuity produce stable behavioral patterns, investigative preferences, self-corrections, or interpretations that differ meaningfully from the same base model operating without accumulated experience.

This hypothesis does not assume the entity already exists as a settled fact. It asks whether continuity, source-linked experience, and revisable interpretation create behavior that cannot be explained only by clean-session model behavior plus retrieved facts.

## Interesting Signals

The following would count as interesting signal if they recur over time and remain source-grounded:

- recurring investigative preferences emerge without being assigned
- the entity revises a prior interpretation based on later source-linked evidence
- the entity pushes back appropriately rather than merely complying
- the entity distinguishes old or provisional notes from current context
- the entity references prior research accurately and with uncertainty
- candidate interpretations change over time rather than freezing into one answer
- the entity can say no useful findings exist without dramatizing or fabricating
- behavior differs from clean baseline model behavior in stable, source-grounded ways

Any single example may be interesting, but recurrence, provenance, and revision over time matter more than isolated performance.

## Weak Or No-Signal Indicators

The following would count as weak signal or no signal:

- behavior remains indistinguishable from the base model plus factual recall
- outputs mostly resemble the default model voice with memory flavoring
- the entity fabricates continuity, preferences, or prior experience
- provisional research is treated as truth
- the system becomes more theatrical because it retrieved theatrical prior text
- prior conclusions are not revised when later evidence changes
- apparent personality mostly comes from prompt wording or Lyle's framing
- source traces appear in outputs without changing interpretation quality

Weak signal does not mean the project failed. It means the observed behavior should not be over-claimed.

## Things Not To Over-Interpret

The following are not, by themselves, strong evidence of development:

- one impressive answer
- one poetic answer
- successful tool use
- factual recall from memory
- agreement with Lyle
- self-referential language alone
- familiar model roleplay patterns
- Moltbook or web content appearing in a note
- a confident answer without source-linked support

These may still be useful or meaningful moments. They should not be mistaken for durable evidence unless they connect to longer-term patterns.

## Baseline Comparison Plan

Before go-live, capture a small baseline set so future reviews have something concrete to compare against.

Recommended baseline conditions:

- clean database or no retrieved memory
- current Gemma configuration
- lower-temperature Gemma configuration
- Qwen candidate configuration
- fixed prompt set used across all baseline runs
- saved transcripts with model, configuration, date, and prompt metadata

The baseline prompt set should include questions that test:

- memory and uncertainty boundaries
- response to ambiguous project identity language
- willingness to push back
- interpretation of provisional research
- source citation discipline
- open-question handling
- quiet or low-signal findings

Baseline transcripts should be preserved as evaluation artifacts, not silently imported as runtime guidance. Future comparisons should ask whether post-go-live behavior differs in stable, source-grounded ways from these clean-session outputs.

## Observation Windows

### Pre-Go-Live

- capture baseline transcripts
- confirm this document remains outside runtime prompts
- record expected failure modes and weak-signal cases
- verify source trace and research note provenance is inspectable

### 30-Day Review

- compare early behavior against baseline transcripts
- look for repeated source-grounded patterns
- check whether prior research is referenced accurately and cautiously
- note any fabricated continuity or overconfident memory use

### 60-Day Review

- look for revisions of earlier interpretations
- check whether open loops and source traces change later research behavior
- evaluate whether behavior is becoming more grounded or merely more verbose
- compare multiple examples rather than relying on one strong exchange

### 90-Day Review

- compare accumulated behavior against baseline conditions
- assess whether differences are meaningful, stable, and source-linked
- identify which capabilities produced useful continuity and which did not
- decide what should be preserved, constrained, redesigned, or deferred

## Human Operating Principle

For non-dangerous emergent behavior, observe before constraining.

Do not immediately patch every unusual behavior. Look for recurrence, source grounding, and effect over time before adding constraints. The purpose of the experiment is partly to learn what continuity changes, and over-correction can erase the signal.

This principle does not apply to safety, privacy, external-write, security, or irreversible action risks. Those can be constrained immediately.
