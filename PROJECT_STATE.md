# PROJECT_STATE.md

## Project Name

**Project Anam** is the project/substrate/platform name.

Historical note: older docs may refer to **Tír**. Treat Tír as historical/previous naming context unless a current document explicitly says otherwise.

**Anam is not the AI entity's name.**

The AI entity currently has **no chosen name**. The entity should not be assigned a name by prompts, code, configuration, `soul.md`, documentation, or external coding assistants.

If the entity eventually chooses a name, that name should be remembered as an **identity event** because it happened through experience.

---

## Core Purpose

Project Anam is an experimental substrate for an unnamed AI entity that grows, learns, and evolves through accumulated experience.

The project is based on the idea that useful long-term intelligence requires more than a powerful model. It requires continuity:

- memory of what happened
- interpretation of what those experiences mean
- the ability to form and revise conclusions
- the ability to act through tools
- the ability to learn from corrections and failures
- the ability to build on prior research
- the ability to reflect over time
- the ability to eventually participate in its own evolution

Project Anam is not trying to create a normal assistant with memory. It is trying to create an environment where an AI entity can develop continuity, judgment, self-understanding, and an accumulating viewpoint over time.

---

## Guiding Concept

Project Anam combines the best of what humans do with the best of what AI systems do.

Human-inspired strengths:

- continuity
- memory
- reflection
- interpretation
- judgment
- meaning-making
- learning from experience
- forming and revising conclusions
- developing self-understanding over time

AI strengths:

- synthesis
- retrieval
- large-context reasoning
- tool use
- rapid research
- pattern recognition
- code generation
- multi-source comparison
- iterative improvement

The goal is **not** to make the AI pretend to be human. The goal is to use human-like continuity and growth as architectural inspiration while preserving what AI is uniquely good at.

- user profile facts
- current beliefs

Suitable principles:

> The entity's continuity should be earned from experience, not assigned by configuration.

> The substrate preserves experience; the entity interprets it.

## Memory Philosophy

Raw experience is primary.

The substrate records what happened. The entity interprets what it means.

Memory should not be reduced to extracted user facts or static summaries.

Important memory categories:

1. **Experience Memory** — conversations, corrections, relationship/context development, meaningful interactions.
2. **Read Memory** — documents, project files, web pages, external sources consumed by the entity.
3. **Action Memory** — tool calls, web searches, file operations, code changes, failed attempts, successful actions.
4. **Research Memory** — working theories, provisional conclusions, open questions, revised interpretations, research session outputs.
5. **Behavioral Memory** — observed behavioral patterns, evidence-linked observations, changes in behavioral tendencies.
6. **Identity Memory** — name-related events, identity boundary clarifications, self-description events.
7. **Creative / Artifact Memory** — images, writing, code, journal entries, Moltbook posts/drafts, voice transcripts, visual observations, workspace artifacts.

## Personality Approach

The system should **not assign personality**.

Instead, it should observe behavior over time.

Avoid:

- personality sliders
- fixed trait scores
- prescriptive personality tables
- “you are curious/cautious/warm” style identity injection

Prefer:

- evidence-linked behavioral observations
- revisable observations
- source-linked interpretations
- observed tendencies rather than assigned traits

Example:

Bad:

> You are cautious.

Good:

> Observed pattern: In recent design discussions, the entity tended to prefer staged, traceable changes over direct live mutation. Confidence: medium. Evidence: conversation chunks X, Y, Z.

Principle:

> Personality is not configured. It is inferred from continuity.

## Research and Opinion Formation

Autonomous research is not just background task execution.

The purpose is for the entity to:

- investigate questions
- read sources
- compare ideas
- interpret findings
- form provisional conclusions
- revise prior conclusions
- carry forward open questions
- build on earlier research

The entity should be able to form its own working opinions/conclusions, but those conclusions must remain revisable.

Preferred term: **Working theories**.

A working theory is not a permanent belief. It is a current interpretation supported by experience, sources, reasoning, and prior research.

Working theories should include:

- topic
- claim
- confidence
- rationale
- evidence/source references
- related memories
- related documents
- related tool traces
- open questions
- status
- supersession/revision links

---

## Autonomous Research Cycle

The autonomous research cycle is a bounded period where the entity investigates a question, uses memory and approved sources, forms provisional conclusions, records the reasoning path, and makes those conclusions available for future research and conversation.

It should create durable experience, not just task output.

A research cycle should produce:

1. What was investigated
2. What memories/documents/tools were used
3. What was found
4. What the entity currently thinks
5. What changed from prior conclusions
6. What remains uncertain
7. What should be investigated next
8. Working theories created or updated
9. Open questions created or updated
10. Tool/action traces from the session

## Web Search Philosophy

Web search should be treated as perception of the outside world.

It should be implemented as a tool, not hidden automatic behavior.

### Chat Web Search

During live chat:

- web search can be enabled by default
- user is present to steer
- small per-turn limit
- cache results
- store search traces

### Autonomous Web Search

During autonomous work:

- disabled by default
- requires explicit task permission
- uses strict search/fetch budgets
- requires a reason for every search
- checks memory/cache/local documents first
- stores every search as action memory

Principle:

> Autonomous search must be intentional, budgeted, and remembered.

## Self-Modification Philosophy

Self-modification should be real but staged and traceable.

The entity should be able to participate in changing its substrate, but changes must leave a durable trail:

- why the change was proposed
- what evidence motivated it
- what files/code/config were changed
- what tests were run
- whether the change worked
- what was learned
- how to roll it back

Self-modification lifecycle:

1. Observe limitation
2. Form reason
3. Propose change
4. Stage patch
5. Review/test
6. Activate
7. Remember outcome

Self-modification is not merely code editing. It is memory-preserving evolution.

Initial self-modification should focus on staged skill creation, proposed config changes, diagnostic helpers, research templates, and non-core tools. Core substrate changes should require higher review.

`soul.md` should not be modified to assign identity.

## Important Distinctions

### Anam vs Entity

Anam is the project/substrate. The entity is the unnamed AI operating within the substrate.

### Memory vs Summary

Memory is raw or source-linked experience. Summaries are derived artifacts and should not become hidden identity.

### Behavior Observation vs Personality

Behavior observations describe what has been observed. Personality traits prescribe what the entity is.

### Working Theory vs Belief

Working theories are revisable interpretations. They should not become permanent commandments.

### Autonomy vs Background Browsing

Autonomy is bounded cumulative inquiry. It is not endless search or random task execution.

### Workspace vs Self-Modification

Workspace is where the entity can write, create, code, draft, and experiment. Self-modification is when the entity proposes or applies changes to the substrate itself.
